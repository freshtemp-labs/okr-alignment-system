## 第一章：文档概述

### 1.1 文档目的

本文档是 **OKR对齐管理系统（OKR Alignment Management System，简称 OAMS）** 的产品需求文档（PRD），旨在为工程团队、设计团队、测试团队及产品团队提供统一的产品开发基准。

本文档的具体目的包括：

1. **定义产品范围**：明确 V1.0 MVP 及后续版本的功能边界，确保团队对交付物有一致认知。
2. **指导开发实施**：每个功能需求均包含功能描述、前置条件、主流程、异常流程、界面描述及验收标准，开发人员可直接据此进行技术实现。
3. **统一验收基准**：为 QA 团队提供明确的测试依据，每个需求点配有可验证的验收标准。
4. **对齐多方期望**：作为产品经理、项目经理、设计师和开发工程师之间的契约文档，降低沟通成本。
5. **支持项目规划**：为项目经理提供任务拆分和排期的输入依据。

**目标读者**：
- iOS/macOS 开发工程师
- UI/UX 设计师
- QA 测试工程师
- 产品经理
- 技术架构师
- 项目管理人员

### 1.2 产品愿景

> **愿景声明**：让每一位组织成员都能清晰地看见自己的工作如何支撑公司战略目标，并通过实时级联反馈激发持续进步。

OAMS 致力于解决以下核心痛点：

| 痛点 | 现状 | OAMS 解决方案 |
|------|------|--------------|
| 目标对齐不透明 | 员工不清楚个人工作与 company OKR 的关联 | 树状可视化，层层穿透，一目了然 |
| 进度反馈滞后 | 季度末才发现目标偏离 | 实时级联计算，任何 KR 更新即刻反映到顶层 |
| 手动汇总低效 | 管理层花费大量时间收集和汇总进度 | 自动级联汇总，叶子 KR 更新自动向上传播 |
| 缺乏责任感 | 目标归属不清晰，无人对结果负责 | 每个节点明确 Owner，责任到人 |
| 数据孤岛 | OKR 数据散落在多个文档/工具中 | 统一平台，支持导入导出，iCloud 同步 |

**核心价值主张**：
- **可视化对齐**：通过树状结构直观展示企业 Objective → 企业 KR → 个人 Objective → 个人 KR 的完整对齐链路。
- **智能级联**：叶子 KR 的进度变更自动触发级联计算，实时更新所有父节点的聚合进度。
- **跨平台体验**：原生 macOS/iOS 应用，充分利用 Apple 生态优势，支持 iCloud 数据同步。
- **隐私优先**：企业级数据加密，本地优先存储，可选 iCloud 同步。

### 1.3 术语定义

| 术语 | 英文全称 | 定义 |
|------|----------|------|
| OKR | Objectives and Key Results | 目标与关键结果法，一种目标管理框架 |
| O | Objective | 目标，描述"我们想要达成什么"，定性、鼓舞人心 |
| KR | Key Result | 关键结果，描述"如何衡量目标达成"，定量、可衡量 |
| OAMS | OKR Alignment Management System | OKR对齐管理系统，本文档所指产品 |
| 企业级 OKR | Enterprise OKR | 公司/部门层面的 OKR，用金色标识 |
| 个人级 OKR | Personal OKR | 个人层面的 OKR，用蓝色标识 |
| 级联计算 | Cascade Calculation | 子节点进度自动汇总到父节点的计算机制 |
| 叶子 KR | Leaf KR | 树状结构中最底层的关键结果，无子节点，直接输入 current/target 值 |
| Owner | Owner | 每个 OKR 节点的负责人，对该节点的结果负责 |
| 对齐 | Alignment | 个人/团队 OKR 向上支撑企业 OKR 的关联关系 |
| 周期 | Cycle | OKR 的执行周期，通常为季度（Q1/Q2/Q3/Q4） |
| MVP | Minimum Viable Product | 最小可行产品，包含核心功能的第一版 |

## 第二章：产品范围

### 2.1 产品定位

OAMS 是一款面向企业组织的 **原生 Apple 平台 OKR 管理应用**，主打 **可视化对齐** 和 **智能级联** 两大核心能力。

**产品定位矩阵**：

| 维度 | 定位 |
|------|------|
| **目标市场** | 中小型企业（50-1000人）、创业团队、部门级管理 |
| **平台策略** | Apple 生态优先（macOS + iOS），原生 SwiftUI 开发 |
| **定价模式** | 基础版免费（限 1 个周期/3 人），Pro 版订阅制 |
| **核心差异化** | 原生级联可视化 + Apple 生态深度集成（iCloud/Widget/Shortcuts） |
| **竞品差异** | 相比网页版 OKR 工具（如飞书OKR、北极星），提供原生体验和本地数据隐私 |

**核心价值主张**：
> "让你的每一次进步，都被组织看见。"

## 第三章：用户故事

### 3.1 企业管理员故事

#### 故事 US-ADMIN-01：创建企业级 Objective
> **作为** 企业管理员（CEO/CTO/总监），  
> **我想要** 创建一个企业级 Objective，  
> **以便** 为组织设定清晰的战略方向。
>
> **验收标准**：
> - [ ] 可以通过界面创建新的企业级 Objective
> - [ ] 可以输入 Objective 标题（最多 200 字符）
> - [ ] 可以设置 Objective 所属周期（Q1/Q2/Q3/Q4）
> - [ ] 可以指定 Objective 的 Owner
> - [ ] 创建后自动出现在 OKR 树顶端
> - [ ] 创建时间不超过 2 秒

#### 故事 US-ADMIN-02：创建企业级 Key Result
> **作为** 企业管理员，  
> **我想要** 为某个企业 Objective 创建 Key Result，  
> **以便** 量化衡量目标的达成情况。
>
> **验收标准**：
> - [ ] 可以在企业 Objective 下添加 KR
> - [ ] 可以输入 KR 标题（最多 200 字符）
> - [ ] 可以选择 KR 的类型：容器型（有子节点）或度量型（叶子节点）
> - [ ] 度量型 KR 可以设置 target 值和单位
> - [ ] 可以指定 KR 的 Owner（默认为 Objective 的 Owner）
> - [ ] 创建后企业 Objective 下显示该 KR

#### 故事 US-ADMIN-03：查看全公司 OKR 树
> **作为** 企业管理员，  
> **我想要** 以树状视图查看全公司的 OKR 对齐关系，  
> **以便** 一目了然地掌握战略目标如何层层分解到个人。
>
> **验收标准**：
> - [ ] 可以展开/折叠任意层级的节点
> - [ ] 企业级节点用金色边框标识，个人级用蓝色边框标识
> - [ ] 每个节点显示 Owner 标签
> - [ ] 每个节点显示实时进度条和百分比
> - [ ] 树状结构清晰显示父子对齐关系
> - [ ] 页面加载时间不超过 3 秒（100 个节点以内）

#### 故事 US-ADMIN-04：监控关键 KR 进展
> **作为** 企业管理员，  
> **我想要** 重点关注某些关键 KR 的进展，  
> **以便** 在出现风险时及时干预。
>
> **验收标准**：
> - [ ] 可以将任意 KR 标记为"关注"
> - [ ] 关注的 KR 在 Dashboard 上集中显示
> - [ ] 当关注的 KR 进度更新时收到通知
> - [ ] 可以关注最多 10 个 KR

#### 故事 US-ADMIN-05：导出 OKR 报表
> **作为** 企业管理员，  
> **我想要** 将当前周期的 OKR 数据导出为文件，  
> **以便** 用于管理层汇报和存档。
>
> **验收标准**：
> - [ ] 可以导出为 CSV 格式
> - [ ] 可以导出为 JSON 格式
> - [ ] 导出包含完整的树状结构和对齐关系
> - [ ] 导出文件包含所有节点的进度信息
> - [ ] 导出操作在 5 秒内完成

### 3.2 团队负责人故事

#### 故事 US-LEAD-01：创建个人 Objective 对齐企业 KR
> **作为** 团队负责人，  
> **我想要** 为团队成员创建个人 Objective，并将其对齐到企业 KR 下，  
> **以便** 确保团队工作方向与公司战略一致。
>
> **验收标准**：
> - [ ] 可以在任意企业 KR 下创建个人 Objective
> - [ ] 个人 Objective 自动成为该企业 KR 的子节点
> - [ ] 可以指定该 Objective 的 Owner（团队成员）
> - [ ] 创建后 OKR 树自动更新显示对齐关系
> - [ ] 企业 KR 的进度自动包含该个人 Objective 的贡献

#### 故事 US-LEAD-02：分配个人 KR 给团队成员
> **作为** 团队负责人，  
> **我想要** 为团队成员的个人 Objective 分配具体的 KR，  
> **以便** 明确每个成员的具体交付物。
>
> **验收标准**：
> - [ ] 可以在个人 Objective 下添加叶子 KR
> - [ ] 可以为叶子 KR 设置 currentValue、targetValue 和 unit
> - [ ] 系统自动计算叶子 KR 的进度百分比
> - [ ] KR 的 Owner 默认为 Objective 的 Owner，但可修改
> - [ ] 叶子 KR 的进度变化自动向上级联

#### 故事 US-LEAD-03：查看团队进度汇总
> **作为** 团队负责人，  
> **我想要** 查看我负责的团队的整体进度，  
> **以便** 评估团队绩效和调整资源分配。
>
> **验收标准**：
> - [ ] 可以筛选只看某个团队/个人的 OKR
> - [ ] 显示团队平均完成进度
> - [ ] 显示各成员的进度对比
> - [ ] 显示进度变化趋势
> - [ ] 数据实时更新，无需手动刷新

#### 故事 US-LEAD-04：审批个人 OKR
> **作为** 团队负责人，  
> **我想要** 审批团队成员提交的个人 OKR，  
> **以便** 确保个人 OKR 的质量和对齐度。

### 4.1 OKR 生命周期管理

#### 4.1.1 Objective 创建（FR-OBJ-001）

**功能描述**：允许授权用户创建新的 Objective（目标）节点。Objective 是 OKR 树中的核心节点，代表一个定性目标。

**优先级**：P0  
**关联用户故事**：US-ADMIN-01, US-LEAD-01

**前置条件**：
- 用户已登录并进入 OKR 管理界面
- 用户具有创建 Objective 的权限
- 当前存在活跃的 OKR 周期

**主流程**：
1. 用户点击「新建 Objective」按钮（或按快捷键 Cmd+N）
2. 系统弹出创建表单（Sheet/Modal）
3. 用户填写以下必填项：
   - **标题**（文本，1-200 字符）
   - **范围**（单选：企业级 / 个人级）
   - **所属周期**（下拉选择，默认当前活跃周期）
4. 用户填写以下可选项：
   - **描述**（文本，0-1000 字符）
   - **Owner**（下拉选择用户，默认当前用户）
   - **父节点**（如果在树中选择父节点后创建，自动填充）
5. 用户点击「保存」
6. 系统验证表单数据
7. 系统创建 Objective 记录
8. 系统在 OKR 树中显示新节点
9. 级联引擎自动重新计算受影响路径的进度

**异常流程**：
- **E1：标题为空或超长**
  - 系统高亮标题输入框，显示红色提示："标题不能为空" 或 "标题不能超过 200 字符"
  - 保存按钮置灰，直至输入有效
- **E2：未选择范围**
  - 系统高亮范围选择区域
  - 显示提示："请选择 Objective 范围"
- **E3：网络异常（iCloud 同步场景）**
  - 本地保存成功，iCloud 同步失败时显示离线提示
  - 数据进入待同步队列，网络恢复后自动重试

**界面描述**：
- **macOS**：使用 NSSheet 从窗口底部滑出，左侧显示表单，右侧实时预览在树中的位置
- **iOS**：使用 `.sheet()` 全屏模态，表单使用 `Form` 视图，键盘自适应
- 标题输入框为焦点自动获取
- 字符计数器实时显示（如 "45/200"）

**验收标准**：
- [ ] 用户可以在 3 步以内开始创建 Objective
- [ ] 表单在 200ms 内响应用户输入
- [ ] 保存后新节点在 500ms 内出现在树中
- [ ] 标题长度验证在前端实时完成
- [ ] 创建操作支持键盘快捷键 Cmd+N（macOS）

#### 4.1.3 Objective 删除（FR-OBJ-003）

**功能描述**：允许授权用户删除 Objective 节点。删除操作需处理子节点的级联关系。

**优先级**：P0  
**关联用户故事**：US-ADMIN-01

**主流程**：
1. 用户右键/长按 Objective 节点，选择「删除」
2. 系统弹出确认对话框，显示以下信息：
   - 要删除的节点名称
   - 该节点下的子节点数量
   - 警告提示：子节点将一并删除或需要重新指定父节点
3. 用户确认删除
4. 系统执行删除操作
5. 级联引擎重新计算上级节点进度
6. OKR 树移除被删节点及其子树

**异常流程**：
- **E1：用户取消确认**
  - 对话框关闭，不做任何操作
- **E2：删除根级企业 Objective**

**界面描述**：
- 确认对话框使用系统原生 `.alert()`
- macOS：使用 NSAlert，带警告图标
- iOS：使用系统 Alert，破坏性操作按钮标红

**验收标准**：
- [ ] 删除操作有二次确认
- [ ] 删除后树状结构 500ms 内刷新
- [ ] 根节点删除有额外安全措施
- [ ] 删除操作不可逆，数据不可恢复

#### 4.1.4 Key Result 创建（FR-KR-001）

**功能描述**：在指定 Objective 下创建 Key Result（关键结果）节点。KR 分为容器型（可包含子节点）和度量型（叶子节点，直接追踪进度）。

**优先级**：P0  
**关联用户故事**：US-ADMIN-02, US-LEAD-02

**前置条件**：
- 用户已登录
- 父 Objective 节点已存在且用户有编辑权限

**主流程**：
1. 用户在目标 Objective 上选择「添加 KR」
2. 系统弹出 KR 创建表单
3. 用户填写必填项：
   - **标题**（文本，1-200 字符）
   - **KR 类型**（单选）：
     - 容器型 KR（用于汇总子节点，无 current/target）
     - 度量型 KR（叶子节点，需设置 current/target）
4. 如选择度量型，额外填写：
   - **当前值**（currentValue，数值，默认 0）
   - **目标值**（targetValue，数值，必须 > 0）
   - **单位**（unit，文本，如 "%"、"人"、"fps"，最多 20 字符）
5. 可选填写：
   - **Owner**（默认继承父 Objective 的 Owner）
   - **描述**（0-500 字符）
6. 用户保存，系统创建 KR 节点
7. OKR 树在对应位置显示新 KR
8. 级联引擎重新计算父节点进度

**异常流程**：
- **E1：targetValue 为 0 或负数**
  - 显示提示："目标值必须大于 0"
  - 保存按钮置灰
- **E2：currentValue > targetValue**
  - 允许输入（可超额完成），但进度计算上限为 100%
- **E3：父节点不存在**
  - 提示："父节点已被删除，请重新选择"

**界面描述**：
- KR 类型选择使用分段控件（Segmented Control）
- 选择度量型后，current/target/unit 输入框动画滑入
- 单位输入框提供常用单位快捷选项：%, 人, 个, 次, 元, fps, ms

**验收标准**：
- [ ] KR 创建表单与 Objective 表单样式一致
- [ ] 度量型 KR 的数值输入支持整数和小数
- [ ] KR 类型切换时表单动画过渡自然（300ms）
- [ ] 创建后级联进度在 500ms 内更新

---

#### 4.1.5 Key Result 编辑与进度更新（FR-KR-002）

**功能描述**：编辑 KR 信息，尤其是更新度量型 KR 的 currentValue 以反映最新进度。

**优先级**：P0  
**关联用户故事**：US-USER-02

**主流程 - 快速进度调整**：
1. 用户将鼠标悬停/轻点在叶子 KR 节点上
2. 节点显示操作控件：`[-10%]` `[+10%]` 按钮
3. 用户点击 `+10%` 按钮
4. 系统计算新值：`currentValue = min(targetValue, currentValue + targetValue * 0.1)`
5. 进度条动画过渡到新宽度（500ms ease-out）
6. 级联引擎触发，向上更新所有父节点的聚合进度
7. 百分比数字实时变化

**主流程 - 精确值编辑**：
1. 用户点击 KR 节点的进度区域
2. 弹出编辑面板，显示当前值输入框
3. 用户输入精确数值
4. 确认后更新，级联计算触发

**异常流程**：
- **E1：currentValue < 0**
  - 自动截断为 0
  - 显示提示："当前值不能小于 0"
- **E2：更新期间父节点被删除**
  - 本地保存 currentValue 更新成功
  - 级联计算到已删除节点时静默终止
  - 提示用户："部分上级节点已变更，进度已局部更新"

**界面描述**：
- `±10%` 按钮仅在 hover/轻点时显示（macOS hover / iOS tap）
- 按钮过渡动画：opacity 0 → 1，200ms
- 进度条宽度变化动画：500ms，cubic-bezier(0.4, 0, 0.2, 1)
- iOS 支持触觉反馈（轻触 ±10% 按钮时轻微震动）

**验收标准**：
- [ ] 点击 ±10% 后，currentValue 在 100ms 内更新
- [ ] 进度条动画 500ms 内完成，过渡平滑
- [ ] 级联更新在 500ms 内完成，无明显卡顿
- [ ] iOS 设备有触觉反馈
- [ ] 精确值编辑支持键盘输入和步进器（Stepper）

### 4.2 树状可视化与导航

#### 4.2.1 树状布局渲染（FR-TREE-001）

**功能描述**：以树状结构渲染 OKR 层级关系，支持企业 Objective → 企业 KR → 个人 Objective → 个人 KR 的完整对齐链路展示。

**优先级**：P0  
**关联用户故事**：US-ADMIN-03, US-USER-03

**主流程**：
1. 系统加载当前周期的 OKR 数据
2. 从根节点开始，递归渲染整棵树
3. 每个节点渲染为卡片样式
4. 父子节点之间用连线表示层级关系
5. 同级节点水平排列，用横线连接
6. 父节点到子节点用竖线连接

**渲染规则**：
- 根节点居中显示
- 子节点均匀分布在父节点下方
- 连线使用 2px 实线，颜色 `rgba(255, 255, 255, 0.1)`
- 节点间距：水平 40px，垂直层级间距 60px

**节点卡片规格**：
- 宽度：280px（macOS），240px（iOS）
- 内边距：16px
- 背景：`rgba(255, 255, 255, 0.05)` + backdrop-filter blur(10px)
- 边框：1px solid `rgba(255, 255, 255, 0.1)`
- 圆角：12px
- 阴影：`0 4px 20px rgba(0, 0, 0, 0.2)`

**异常流程**：
- **E1：数据循环引用**
  - 检测到循环时终止渲染该分支
  - 显示错误标记，记录日志
- **E2：节点过多（>200）**
  - 自动启用虚拟化渲染，只渲染可视区域节点
  - 显示节点总数提示

**界面描述**：
- 整个树状区域支持拖拽平移（macOS + iOS）
- macOS 支持触控板双指平移和捏合缩放

### 4.3 级联进度引擎

#### 4.3.1 叶子 KR 进度计算（FR-CALC-001）

**功能描述**：对叶子节点（度量型 Key Result）计算其进度百分比。

**优先级**：P0  
**关联用户故事**：US-USER-02

**计算公式**：
```
progress = (currentValue / targetValue) × 100%
progress = min(100, max(0, progress))
```

**计算规则**：
- 结果四舍五入到 1 位小数（如 "67.3%"）
- 上限为 100%（超额完成不显示 >100%）
- 下限为 0%（currentValue < 0 时按 0 处理）
- 进度变化时触发级联计算

**示例**：
| currentValue | targetValue | unit | 计算进度 | 显示 |
|-------------|-------------|------|----------|------|
| 20 | 100 | % | 20.0% | "20 / 100 %" |
| 5 | 50 | 人 | 10.0% | "5 / 50 人" |
| 30 | 60 | fps | 50.0% | "30 / 60 fps" |
| 110 | 100 | % | 100.0% | "110 / 100 %" |

**验收标准**：
- [ ] 计算结果精确到 1 位小数
- [ ] 结果始终限制在 [0%, 100%] 范围
- [ ] currentValue 更新后 100ms 内完成计算
- [ ] 单位正确显示在进度统计中

---

#### 4.3.2 父节点聚合进度计算（FR-CALC-002）

**功能描述**：对非叶子节点（Objective 和容器型 KR），通过聚合子节点的进度计算自身进度。

**优先级**：P0  
**关联用户故事**：US-ADMIN-03, US-USER-03

**计算公式**：
```
if node.children.length > 0:
    node.progress = average(child.progress for child in node.children)
else:
    node.progress = 0
```

**计算规则**：
- 父节点进度 = 所有直接子节点进度的算术平均值
- 递归计算：从叶子节点开始，逐层向上汇总
- 结果四舍五入到 1 位小数
- 任意叶子节点变化时，触发从该节点到根节点的完整重算

**级联触发时机**：
- 叶子 KR 的 currentValue 更新时
- KR 被创建或删除时
- Objective 被创建或删除时
- 节点父子关系变更时

**计算示例**：
```
企业 O: 打造极具竞争力的商业化工具
└── 企业 KR1: 完成 MVP (父节点)
    ├── 个人 O1: 确保产品体验 (Owner: Alice)
    │   ├── KR1: 完成设计 [20/100 = 20%]
    │   └── KR2: 招募用户 [5/50 = 10%]
    │   └── 平均 = (20 + 10) / 2 = 15.0%
    ├── 个人 O2: 攻克技术难点 (Owner: Bob)
    │   ├── KR3: Native Bridge [0/100 = 0%]
    │   └── KR4: 渲染优化 [30/60 = 50%]
    │   └── 平均 = (0 + 50) / 2 = 25.0%
    └── 企业 KR1 进度 = (15 + 25) / 2 = 20.0%
企业 O 进度 = 20.0%
```

**优化策略**：
- 使用增量计算，只重新计算受影响的路径
- 对大型树（>100 节点）使用异步计算，避免阻塞 UI
- 计算结果缓存，相同输入不再重复计算

**验收标准**：
- [ ] 叶子 KR 更新后，到根节点的级联计算在 500ms 内完成
- [ ] 计算结果精确（与手工计算一致）
- [ ] 100 个节点的完整树计算在 1 秒内完成
- [ ] 级联过程中 UI 保持响应（不卡顿）
- [ ] 并发多个更新时，计算结果最终一致

### 6.1 设计原则与风格指南

#### 6.1.1 设计原则

| 原则 | 描述 | 应用方式 |
|------|------|----------|
| **清晰** | 信息层次清晰，用户一眼找到重点 | 通过颜色、字号、间距建立视觉层级 |
| **高效** | 常用操作快速触达，减少操作步数 | 快捷键、上下文操作、批量操作 |
| **愉悦** | 精美的动效和视觉反馈 | 进度条动画、触觉反馈、过渡动画 |
| **一致** | 与 Apple 设计语言一致 | 使用 SwiftUI 原生组件，遵循 HIG |
| **响应** | 任何操作都有即时反馈 | 按钮状态变化、加载指示器、动画过渡 |

### 6.5 交互动效规格

#### 6.5.1 进度条动画（核心动效）

**触发时机**：叶子 KR 的 currentValue 更新时

**动画参数**：
- 属性：进度条宽度（`width`）
- 持续时间：500ms
- 缓动函数：`cubic-bezier(0.4, 0, 0.2, 1)`（ease-out）
- 级联传播：父节点延迟 100ms 后启动相同动画

**实现要点**：
- 使用 SwiftUI `withAnimation(.easeOut(duration: 0.5))`
- 百分比数字同步使用 `Animatable` 协议实现计数动画
- 动画可被打断（新更新到来时立即跳到新目标值）

#### 6.5.2 级联瀑布动画

**触发时机**：任意叶子节点更新后

**动画序列**：
```
t=0ms:    叶子 KR 进度条开始动画
t=500ms:  叶子 KR 动画完成
t=600ms:  直接父节点进度条开始动画（延迟 100ms）
t=1100ms: 直接父节点动画完成
t=1200ms: 祖父节点进度条开始动画
...以此类推直到根节点
```

**总时长估算**：`500ms + (depth - 1) × 100ms`
- 4 层深度：约 800ms
- 8 层深度：约 1200ms

#### 6.5.3 节点展开/折叠动画

**展开动画**：
- 子树从 `opacity: 0, scale: 0.95` 到 `opacity: 1, scale: 1`
- 持续时间：300ms
- 缓动：spring(response: 0.3, dampingFraction: 0.8)

**折叠动画**：
- 子树从 `opacity: 1, scale: 1` 到 `opacity: 0, scale: 0.95`
- 持续时间：200ms
- 缓动：easeIn

**展开/折叠按钮旋转**：
- 展开：`chevron.right` 旋转 90° 变为 `chevron.down`
- 持续时间：200ms

#### 6.5.4 节点卡片 Hover 效果

- 操作按钮从底部滑入

#### 6.5.5 页面转场动画

description?.setOption(true as NSNumber, forKey: NSPersistentHistoryTrackingKey)
description?.setOption(true as NSNumber, forKey: NSPersistentStoreRemoteChangeNotificationPostOptionKey)

**写入操作**：
- 所有写入通过 `viewContext` 或 `backgroundContext`
- 批量写入使用 `backgroundContext` 避免阻塞 UI
- 写入后自动保存（`container.viewContext.automaticallyMergesChangesFromParent = true`）

**删除操作**：
- 软删除：设置 `status = "archived"`
- 硬删除：仅管理员可执行，记录审计日志
- 级联删除：Core Data 关系配置 `Delete Rule: Cascade`

- 备份文件命名：`OKRModel_backup_YYYYMMDD.sqlite`

#### 7.4.3 同步策略

**自动同步**：
- 默认开启 iCloud 同步

- 使用 NSPersistentCloudKitContainer 自动处理

**同步冲突解决**：
- 默认策略：Last-Write-Wins（以最后修改时间为准）
- 冲突检测：基于 `updatedAt` 时间戳
- V1.5 增加：手动冲突解决界面（三向对比：本地/云端/合并）

**离线策略**：
- 离线时所有操作正常执行，变更记录到本地队列
- 网络恢复后自动同步离线队列
- 同步冲突时按上述策略处理

#### 7.4.4 同步状态指示

**UI 指示**：
- 工具栏显示同步状态图标：

  - ✅ 已同步
  - ⚠️ 同步失败（点击重试）
  - 📴 离线模式
- 最后一次同步时间 tooltip："上次同步：2 分钟前"

#### 7.4.5 同步故障处理

| 故障场景 | 处理方式 | 用户提示 |
|----------|----------|----------|

**职责**：负责 OKR 树中所有节点的级联进度计算

```swift
/// 级联计算引擎协议
protocol CascadeCalculationEngineProtocol {
    
    /// 计算单个节点的进度
    /// - Parameter node: 目标节点
    /// - Returns: 计算后的进度值 (0.0 ~ 100.0)
    func calculateProgress(for node: OKRNode) -> Double
    
    /// 从指定节点向上级联计算到根节点
    /// - Parameter node: 变更的叶子节点
    /// - Returns: 受影响的节点及其新进度值
    func cascadeUp(from node: OKRNode) -> [(nodeID: UUID, newProgress: Double)]
    
    /// 重新计算整棵树的进度
    /// - Parameter rootNode: 根节点
    /// - Returns: 所有节点的进度值映射
    func recalculateTree(rootNode: OKRNode) -> [UUID: Double]
    
    /// 计算指定 KR 的 ±10% 调整值
    /// - Parameters:
    ///   - node: 目标叶子 KR
    ///   - increase: true=+10%, false=-10%
    /// - Returns: 新的 currentValue
    func calculateAdjustedValue(for node: OKRNode, increase: Bool) -> Double
}
```

**计算规则实现**：

```swift
class CascadeCalculationEngine: CascadeCalculationEngineProtocol {
    
    func calculateProgress(for node: OKRNode) -> Double {
        // 1. 如果有子节点，递归计算子节点后取平均
        if let children = node.children, !children.isEmpty {
            let totalProgress = children.reduce(0.0) { sum, child in
                return sum + calculateProgress(for: child)
            }
            return totalProgress / Double(children.count)
        }
        
        // 2. 如果是度量型 KR（叶子节点），计算百分比
        if node.type == .keyResult,
           let current = node.currentValue,
           let target = node.targetValue,
           target > 0 {
            let progress = (current / target) * 100.0
            return min(100.0, max(0.0, progress))
        }
        
        // 3. 空 Objective，返回 0
        return 0.0
    }
    
    func calculateAdjustedValue(for node: OKRNode, increase: Bool) -> Double {
        guard let target = node.targetValue,
              let current = node.currentValue else {
            return 0.0
        }
        
        let delta = target * 0.1
        let newValue = increase ? current + delta : current - delta
        
        // 限制在 [0, target] 范围
        return min(target, max(0.0, newValue))
    }
}
```

        return true

    case (.enterpriseAdmin, .createObjective(.personal)):

        return true
    case (.enterpriseAdmin, .manageCycles),
         (.enterpriseAdmin, .manageUsers):
        return true

    case (.teamLead, .createObjective(.personal)):

### 9.1 技术风险

#### 9.1.1 级联计算性能风险

| 风险项 | 级联计算在大规模数据下可能出现性能瓶颈 |
|--------|--------------------------------------|
| **风险等级** | 🔴 高 |
| **影响范围** | 核心功能（V1.0） |
| **风险描述** | 当 OKR 树节点数量超过 500、层级深度超过 10 层时，级联计算可能导致 UI 卡顿。叶子节点频繁更新时，重复的级联计算会消耗大量 CPU 资源。 |
| **触发条件** | 1. 节点数量 > 500<br>2. 树深度 > 10 层<br>3. 用户在短时间内连续更新多个叶子 KR |
| **应对措施** | 1. **增量计算**：仅重新计算受影响的路径，而非整棵树<br>2. **异步计算**：将级联计算放到后台线程，通过 `@MainActor` 更新 UI<br>3. **计算防抖**：500ms 内的多次更新合并为一次计算<br>4. **缓存机制**：缓存中间计算结果，避免重复计算<br>5. **性能测试**：在 V1.0 发布前进行 1000 节点压力测试 |
| **责任人** | 技术架构师 |
| **监控指标** | 级联计算耗时 < 1s（1000 节点） |

#### 9.1.2 iCloud 同步冲突风险

| 风险项 | iCloud/CloudKit 同步可能出现数据冲突和丢失 |

| **风险等级** | 🔴 高 |

| **触发条件** | 1. 开发人力不足<br>2. 平台差异导致大量适配工作 |
| **应对措施** | 1. **优先 iOS**：V1.0 优先完成 iOS 端，macOS 端在 iOS 稳定后跟进<br>2. **代码复用**：最大化共享 ViewModel 和业务逻辑<br>3. **外部资源**：必要时引入外包资源处理特定平台功能<br>4. **功能裁剪**：MVP 阶段裁剪非核心平台特性 |
| **责任人** | 项目经理 |
| **监控指标** | 按里程碑准时交付率 > 80% |

#### 9.2.3 Apple 审核风险

| 风险项 | App Store / Mac App Store 审核可能被拒绝 |
|--------|------------------------------------------|
| **风险等级** | 🟢 低 |
| **影响范围** | 产品发布（V1.0） |
| **风险描述** | 应用可能因不符合 App Store 审核指南而被拒绝。例如：iCloud 同步功能需要正确的 entitlements 配置；沙盒外的文件访问需要用户授权。 |
| **触发条件** | 1. CloudKit entitlements 配置错误<br>2. 沙盒违规<br>3. 未提供测试账户 |
| **应对措施** | 1. **预审自查**：发布前对照审核指南自查<br>2. **TestFlight 先行**：先通过 TestFlight 分发内测<br>3. **审核指南跟踪**：关注 Apple 审核指南更新<br>4. **备用方案**：如被拒绝，准备快速修复版本 |
| **责任人** | iOS/macOS 开发工程师 |
| **监控指标** | 首次审核通过率 100% |

#### 9.2.4 隐私合规风险

| 风险项 | 数据隐私法规（GDPR、个人信息保护法）可能导致合规问题 |
|--------|----------------------------------------------------|
| **风险等级** | 🟡 中 |
| **影响范围** | 产品合规（V1.0+） |
| **风险描述** | OKR 数据包含员工的工作目标和绩效信息，属于敏感工作数据。如果处理不当，可能违反数据隐私法规。 |
| **触发条件** | 1. 数据未加密存储<br>2. 用户无法导出/删除自己的数据<br>3. 数据跨境传输未告知 |
| **应对措施** | 1. **本地优先**：数据默认本地存储，用户可控<br>2. **数据可导出**：提供完整的数据导出功能<br>3. **数据可删除**：用户可完全删除自己的数据<br>4. **隐私政策**：提供清晰的隐私政策说明<br>5. **加密存储**：本地数据库使用 AES-256 加密 |
| **责任人** | 产品经理 + 法务 |
| **监控指标** | 无用户隐私投诉 |

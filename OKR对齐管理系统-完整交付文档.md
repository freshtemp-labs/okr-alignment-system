## 第一章：文档概述

### 1.1 文档目的

本文档是 **OKR对齐管理系统（OKR Alignment Management System，简称 OAMS）** 的产品需求文档（PRD），旨在为工程团队、设计团队、测试团队及产品团队提供统一的产品开发基准。

本文档的具体目的包括：

1. **定义产品范围**：明确 V1.0 MVP 及后续版本的功能边界，确保团队对交付物有一致认知。
2. **指导开发实施**：每个功能需求均包含功能描述、前置条件、主流程、异常流程、界面描述及验收标准，开发人员可直接据此进行技术实现。
3. **统一验收基准**：为 QA 团队提供明确的测试依据，每个需求点配有可验证的验收标准。
4. **对齐多方期望**：作为产品经理、项目经理、设计师和开发工程师之间的契约文档，降低沟通成本。
5. **支持项目规划**：为项目经理提供任务拆分和排期的输入依据。

**目标读者**：
- iOS/macOS 开发工程师
- UI/UX 设计师
- QA 测试工程师
- 产品经理
- 技术架构师
- 项目管理人员

### 1.2 产品愿景

> **愿景声明**：让每一位组织成员都能清晰地看见自己的工作如何支撑公司战略目标，并通过实时级联反馈激发持续进步。

OAMS 致力于解决以下核心痛点：

| 痛点 | 现状 | OAMS 解决方案 |
|------|------|--------------|
| 目标对齐不透明 | 员工不清楚个人工作与 company OKR 的关联 | 树状可视化，层层穿透，一目了然 |
| 进度反馈滞后 | 季度末才发现目标偏离 | 实时级联计算，任何 KR 更新即刻反映到顶层 |
| 手动汇总低效 | 管理层花费大量时间收集和汇总进度 | 自动级联汇总，叶子 KR 更新自动向上传播 |
| 缺乏责任感 | 目标归属不清晰，无人对结果负责 | 每个节点明确 Owner，责任到人 |
| 数据孤岛 | OKR 数据散落在多个文档/工具中 | 统一平台，支持导入导出，iCloud 同步 |

**核心价值主张**：
- **可视化对齐**：通过树状结构直观展示企业 Objective → 企业 KR → 个人 Objective → 个人 KR 的完整对齐链路。
- **智能级联**：叶子 KR 的进度变更自动触发级联计算，实时更新所有父节点的聚合进度。
- **跨平台体验**：原生 macOS/iOS 应用，充分利用 Apple 生态优势，支持 iCloud 数据同步。
- **隐私优先**：企业级数据加密，本地优先存储，可选 iCloud 同步。

### 1.3 术语定义

| 术语 | 英文全称 | 定义 |
|------|----------|------|
| OKR | Objectives and Key Results | 目标与关键结果法，一种目标管理框架 |
| O | Objective | 目标，描述"我们想要达成什么"，定性、鼓舞人心 |
| KR | Key Result | 关键结果，描述"如何衡量目标达成"，定量、可衡量 |
| OAMS | OKR Alignment Management System | OKR对齐管理系统，本文档所指产品 |
| 企业级 OKR | Enterprise OKR | 公司/部门层面的 OKR，用金色标识 |
| 个人级 OKR | Personal OKR | 个人层面的 OKR，用蓝色标识 |
| 级联计算 | Cascade Calculation | 子节点进度自动汇总到父节点的计算机制 |
| 叶子 KR | Leaf KR | 树状结构中最底层的关键结果，无子节点，直接输入 current/target 值 |
| Owner | Owner | 每个 OKR 节点的负责人，对该节点的结果负责 |
| 对齐 | Alignment | 个人/团队 OKR 向上支撑企业 OKR 的关联关系 |
| 周期 | Cycle | OKR 的执行周期，通常为季度（Q1/Q2/Q3/Q4） |
| MVP | Minimum Viable Product | 最小可行产品，包含核心功能的第一版 |

## 第二章：产品范围

### 2.1 产品定位

OAMS 是一款面向企业组织的 **原生 Apple 平台 OKR 管理应用**，主打 **可视化对齐** 和 **智能级联** 两大核心能力。

**产品定位矩阵**：

| 维度 | 定位 |
|------|------|
| **目标市场** | 中小型企业（50-1000人）、创业团队、部门级管理 |
| **平台策略** | Apple 生态优先（macOS + iOS），原生 SwiftUI 开发 |
| **定价模式** | 基础版免费（限 1 个周期/3 人），Pro 版订阅制 |
| **核心差异化** | 原生级联可视化 + Apple 生态深度集成（iCloud/Widget/Shortcuts） |
| **竞品差异** | 相比网页版 OKR 工具（如飞书OKR、北极星），提供原生体验和本地数据隐私 |

**核心价值主张**：
> "让你的每一次进步，都被组织看见。"

## 第三章：用户故事

### 3.1 企业管理员故事

#### 故事 US-ADMIN-01：创建企业级 Objective
> **作为** 企业管理员（CEO/CTO/总监），  
> **我想要** 创建一个企业级 Objective，  
> **以便** 为组织设定清晰的战略方向。
>
> **验收标准**：
> - [ ] 可以通过界面创建新的企业级 Objective
> - [ ] 可以输入 Objective 标题（最多 200 字符）
> - [ ] 可以设置 Objective 所属周期（Q1/Q2/Q3/Q4）
> - [ ] 可以指定 Objective 的 Owner
> - [ ] 创建后自动出现在 OKR 树顶端
> - [ ] 创建时间不超过 2 秒

#### 故事 US-ADMIN-02：创建企业级 Key Result
> **作为** 企业管理员，  
> **我想要** 为某个企业 Objective 创建 Key Result，  
> **以便** 量化衡量目标的达成情况。
>
> **验收标准**：
> - [ ] 可以在企业 Objective 下添加 KR
> - [ ] 可以输入 KR 标题（最多 200 字符）
> - [ ] 可以选择 KR 的类型：容器型（有子节点）或度量型（叶子节点）
> - [ ] 度量型 KR 可以设置 target 值和单位
> - [ ] 可以指定 KR 的 Owner（默认为 Objective 的 Owner）
> - [ ] 创建后企业 Objective 下显示该 KR

#### 故事 US-ADMIN-03：查看全公司 OKR 树
> **作为** 企业管理员，  
> **我想要** 以树状视图查看全公司的 OKR 对齐关系，  
> **以便** 一目了然地掌握战略目标如何层层分解到个人。
>
> **验收标准**：
> - [ ] 可以展开/折叠任意层级的节点
> - [ ] 企业级节点用金色边框标识，个人级用蓝色边框标识
> - [ ] 每个节点显示 Owner 标签
> - [ ] 每个节点显示实时进度条和百分比
> - [ ] 树状结构清晰显示父子对齐关系
> - [ ] 页面加载时间不超过 3 秒（100 个节点以内）

#### 故事 US-ADMIN-04：监控关键 KR 进展
> **作为** 企业管理员，  
> **我想要** 重点关注某些关键 KR 的进展，  
> **以便** 在出现风险时及时干预。
>
> **验收标准**：
> - [ ] 可以将任意 KR 标记为"关注"
> - [ ] 关注的 KR 在 Dashboard 上集中显示
> - [ ] 当关注的 KR 进度更新时收到通知
> - [ ] 可以关注最多 10 个 KR

#### 故事 US-ADMIN-05：导出 OKR 报表
> **作为** 企业管理员，  
> **我想要** 将当前周期的 OKR 数据导出为文件，  
> **以便** 用于管理层汇报和存档。
>
> **验收标准**：
> - [ ] 可以导出为 CSV 格式
> - [ ] 可以导出为 JSON 格式
> - [ ] 导出包含完整的树状结构和对齐关系
> - [ ] 导出文件包含所有节点的进度信息
> - [ ] 导出操作在 5 秒内完成

### 3.2 团队负责人故事

#### 故事 US-LEAD-01：创建个人 Objective 对齐企业 KR
> **作为** 团队负责人，  
> **我想要** 为团队成员创建个人 Objective，并将其对齐到企业 KR 下，  
> **以便** 确保团队工作方向与公司战略一致。
>
> **验收标准**：
> - [ ] 可以在任意企业 KR 下创建个人 Objective
> - [ ] 个人 Objective 自动成为该企业 KR 的子节点
> - [ ] 可以指定该 Objective 的 Owner（团队成员）
> - [ ] 创建后 OKR 树自动更新显示对齐关系
> - [ ] 企业 KR 的进度自动包含该个人 Objective 的贡献

#### 故事 US-LEAD-02：分配个人 KR 给团队成员
> **作为** 团队负责人，  
> **我想要** 为团队成员的个人 Objective 分配具体的 KR，  
> **以便** 明确每个成员的具体交付物。
>
> **验收标准**：
> - [ ] 可以在个人 Objective 下添加叶子 KR
> - [ ] 可以为叶子 KR 设置 currentValue、targetValue 和 unit
> - [ ] 系统自动计算叶子 KR 的进度百分比
> - [ ] KR 的 Owner 默认为 Objective 的 Owner，但可修改
> - [ ] 叶子 KR 的进度变化自动向上级联

#### 故事 US-LEAD-03：查看团队进度汇总
> **作为** 团队负责人，  
> **我想要** 查看我负责的团队的整体进度，  
> **以便** 评估团队绩效和调整资源分配。
>
> **验收标准**：
> - [ ] 可以筛选只看某个团队/个人的 OKR
> - [ ] 显示团队平均完成进度
> - [ ] 显示各成员的进度对比

### 4.1 OKR 生命周期管理

#### 4.1.1 Objective 创建（FR-OBJ-001）

**功能描述**：允许授权用户创建新的 Objective（目标）节点。Objective 是 OKR 树中的核心节点，代表一个定性目标。

**优先级**：P0  
**关联用户故事**：US-ADMIN-01, US-LEAD-01

**前置条件**：
- 用户已登录并进入 OKR 管理界面
- 用户具有创建 Objective 的权限
- 当前存在活跃的 OKR 周期

**主流程**：
1. 用户点击「新建 Objective」按钮（或按快捷键 Cmd+N）
2. 系统弹出创建表单（Sheet/Modal）
3. 用户填写以下必填项：
   - **标题**（文本，1-200 字符）
   - **范围**（单选：企业级 / 个人级）
   - **所属周期**（下拉选择，默认当前活跃周期）
4. 用户填写以下可选项：
   - **描述**（文本，0-1000 字符）
   - **Owner**（下拉选择用户，默认当前用户）
   - **父节点**（如果在树中选择父节点后创建，自动填充）
5. 用户点击「保存」
6. 系统验证表单数据
7. 系统创建 Objective 记录
8. 系统在 OKR 树中显示新节点
9. 级联引擎自动重新计算受影响路径的进度

**异常流程**：
- **E1：标题为空或超长**
  - 系统高亮标题输入框，显示红色提示："标题不能为空" 或 "标题不能超过 200 字符"
  - 保存按钮置灰，直至输入有效
- **E2：未选择范围**
  - 系统高亮范围选择区域
  - 显示提示："请选择 Objective 范围"
- **E3：网络异常（iCloud 同步场景）**
  - 本地保存成功，iCloud 同步失败时显示离线提示
  - 数据进入待同步队列，网络恢复后自动重试

**界面描述**：
- **macOS**：使用 NSSheet 从窗口底部滑出，左侧显示表单，右侧实时预览在树中的位置
- **iOS**：使用 `.sheet()` 全屏模态，表单使用 `Form` 视图，键盘自适应
- 标题输入框为焦点自动获取
- 字符计数器实时显示（如 "45/200"）

**验收标准**：
- [ ] 用户可以在 3 步以内开始创建 Objective
- [ ] 表单在 200ms 内响应用户输入
- [ ] 保存后新节点在 500ms 内出现在树中
- [ ] 标题长度验证在前端实时完成
- [ ] 创建操作支持键盘快捷键 Cmd+N（macOS）

#### 4.1.3 Objective 删除（FR-OBJ-003）

**功能描述**：允许授权用户删除 Objective 节点。删除操作需处理子节点的级联关系。

**优先级**：P0  
**关联用户故事**：US-ADMIN-01

**主流程**：
1. 用户右键/长按 Objective 节点，选择「删除」
2. 系统弹出确认对话框，显示以下信息：
   - 要删除的节点名称
   - 该节点下的子节点数量

3. 用户确认删除
4. 系统执行删除操作
5. 级联引擎重新计算上级节点进度
6. OKR 树移除被删节点及其子树

**异常流程**：
- **E1：用户取消确认**

- **E2：删除根级企业 Objective**

**界面描述**：
- 确认对话框使用系统原生 `.alert()`
- macOS：使用 NSAlert，带警告图标
- iOS：使用系统 Alert，破坏性操作按钮标红

**验收标准**：
- [ ] 删除操作有二次确认
- [ ] 删除后树状结构 500ms 内刷新
- [ ] 根节点删除有额外安全措施
- [ ] 删除操作不可逆，数据不可恢复

#### 4.1.4 Key Result 创建（FR-KR-001）

**功能描述**：在指定 Objective 下创建 Key Result（关键结果）节点。KR 分为容器型（可包含子节点）和度量型（叶子节点，直接追踪进度）。

**优先级**：P0  
**关联用户故事**：US-ADMIN-02, US-LEAD-02

**前置条件**：
- 用户已登录
- 父 Objective 节点已存在且用户有编辑权限

**主流程**：
1. 用户在目标 Objective 上选择「添加 KR」
2. 系统弹出 KR 创建表单
3. 用户填写必填项：
   - **标题**（文本，1-200 字符）
   - **KR 类型**（单选）：
     - 容器型 KR（用于汇总子节点，无 current/target）
     - 度量型 KR（叶子节点，需设置 current/target）
4. 如选择度量型，额外填写：
   - **当前值**（currentValue，数值，默认 0）
   - **目标值**（targetValue，数值，必须 > 0）
   - **单位**（unit，文本，如 "%"、"人"、"fps"，最多 20 字符）
5. 可选填写：
   - **Owner**（默认继承父 Objective 的 Owner）
   - **描述**（0-500 字符）
6. 用户保存，系统创建 KR 节点
7. OKR 树在对应位置显示新 KR
8. 级联引擎重新计算父节点进度

**异常流程**：
- **E1：targetValue 为 0 或负数**
  - 显示提示："目标值必须大于 0"
  - 保存按钮置灰
- **E2：currentValue > targetValue**
  - 允许输入（可超额完成），但进度计算上限为 100%
- **E3：父节点不存在**
  - 提示："父节点已被删除，请重新选择"

**界面描述**：
- KR 类型选择使用分段控件（Segmented Control）
- 选择度量型后，current/target/unit 输入框动画滑入
- 单位输入框提供常用单位快捷选项：%, 人, 个, 次, 元, fps, ms

**验收标准**：
- [ ] KR 创建表单与 Objective 表单样式一致
- [ ] 度量型 KR 的数值输入支持整数和小数
- [ ] KR 类型切换时表单动画过渡自然（300ms）
- [ ] 创建后级联进度在 500ms 内更新

---

#### 4.1.5 Key Result 编辑与进度更新（FR-KR-002）

**功能描述**：编辑 KR 信息，尤其是更新度量型 KR 的 currentValue 以反映最新进度。

**优先级**：P0  
**关联用户故事**：US-USER-02

**主流程 - 快速进度调整**：
1. 用户将鼠标悬停/轻点在叶子 KR 节点上
2. 节点显示操作控件：`[-10%]` `[+10%]` 按钮
3. 用户点击 `+10%` 按钮
4. 系统计算新值：`currentValue = min(targetValue, currentValue + targetValue * 0.1)`
5. 进度条动画过渡到新宽度（500ms ease-out）
6. 级联引擎触发，向上更新所有父节点的聚合进度
7. 百分比数字实时变化

**主流程 - 精确值编辑**：
1. 用户点击 KR 节点的进度区域
2. 弹出编辑面板，显示当前值输入框
3. 用户输入精确数值
4. 确认后更新，级联计算触发

**异常流程**：
- **E1：currentValue < 0**
  - 自动截断为 0
  - 显示提示："当前值不能小于 0"
- **E2：更新期间父节点被删除**
  - 本地保存 currentValue 更新成功
  - 级联计算到已删除节点时静默终止
  - 提示用户："部分上级节点已变更，进度已局部更新"

**界面描述**：
- `±10%` 按钮仅在 hover/轻点时显示（macOS hover / iOS tap）
- 按钮过渡动画：opacity 0 → 1，200ms
- 进度条宽度变化动画：500ms，cubic-bezier(0.4, 0, 0.2, 1)
- iOS 支持触觉反馈（轻触 ±10% 按钮时轻微震动）

**验收标准**：
- [ ] 点击 ±10% 后，currentValue 在 100ms 内更新
- [ ] 进度条动画 500ms 内完成，过渡平滑
- [ ] 级联更新在 500ms 内完成，无明显卡顿
- [ ] iOS 设备有触觉反馈
- [ ] 精确值编辑支持键盘输入和步进器（Stepper）

### 4.2 树状可视化与导航

#### 4.2.1 树状布局渲染（FR-TREE-001）

**功能描述**：以树状结构渲染 OKR 层级关系，支持企业 Objective → 企业 KR → 个人 Objective → 个人 KR 的完整对齐链路展示。

**优先级**：P0  
**关联用户故事**：US-ADMIN-03, US-USER-03

**主流程**：
1. 系统加载当前周期的 OKR 数据
2. 从根节点开始，递归渲染整棵树
3. 每个节点渲染为卡片样式
4. 父子节点之间用连线表示层级关系
5. 同级节点水平排列，用横线连接
6. 父节点到子节点用竖线连接

**渲染规则**：
- 根节点居中显示
- 子节点均匀分布在父节点下方
- 连线使用 2px 实线，颜色 `rgba(255, 255, 255, 0.1)`
- 节点间距：水平 40px，垂直层级间距 60px

**节点卡片规格**：
- 宽度：280px（macOS），240px（iOS）
- 内边距：16px
- 背景：`rgba(255, 255, 255, 0.05)` + backdrop-filter blur(10px)
- 边框：1px solid `rgba(255, 255, 255, 0.1)`
- 圆角：12px
- 阴影：`0 4px 20px rgba(0, 0, 0, 0.2)`

**异常流程**：
- **E1：数据循环引用**
  - 检测到循环时终止渲染该分支
  - 显示错误标记，记录日志

### 4.3 级联进度引擎

#### 4.3.1 叶子 KR 进度计算（FR-CALC-001）

**功能描述**：对叶子节点（度量型 Key Result）计算其进度百分比。

**优先级**：P0  
**关联用户故事**：US-USER-02

**计算公式**：
```
progress = (currentValue / targetValue) × 100%
progress = min(100, max(0, progress))
```

**计算规则**：
- 结果四舍五入到 1 位小数（如 "67.3%"）
- 上限为 100%（超额完成不显示 >100%）
- 下限为 0%（currentValue < 0 时按 0 处理）
- 进度变化时触发级联计算

**示例**：
| currentValue | targetValue | unit | 计算进度 | 显示 |
|-------------|-------------|------|----------|------|
| 20 | 100 | % | 20.0% | "20 / 100 %" |
| 5 | 50 | 人 | 10.0% | "5 / 50 人" |
| 30 | 60 | fps | 50.0% | "30 / 60 fps" |
| 110 | 100 | % | 100.0% | "110 / 100 %" |

**验收标准**：
- [ ] 计算结果精确到 1 位小数
- [ ] 结果始终限制在 [0%, 100%] 范围
- [ ] currentValue 更新后 100ms 内完成计算
- [ ] 单位正确显示在进度统计中

---

#### 4.3.2 父节点聚合进度计算（FR-CALC-002）

**功能描述**：对非叶子节点（Objective 和容器型 KR），通过聚合子节点的进度计算自身进度。

**优先级**：P0  
**关联用户故事**：US-ADMIN-03, US-USER-03

**计算公式**：
```
if node.children.length > 0:
    node.progress = average(child.progress for child in node.children)
else:
    node.progress = 0
```

**计算规则**：
- 父节点进度 = 所有直接子节点进度的算术平均值
- 递归计算：从叶子节点开始，逐层向上汇总
- 结果四舍五入到 1 位小数
- 任意叶子节点变化时，触发从该节点到根节点的完整重算

**级联触发时机**：
- 叶子 KR 的 currentValue 更新时
- KR 被创建或删除时
- Objective 被创建或删除时
- 节点父子关系变更时

**计算示例**：
```
企业 O: 打造极具竞争力的商业化工具
└── 企业 KR1: 完成 MVP (父节点)
    ├── 个人 O1: 确保产品体验 (Owner: Alice)
    │   ├── KR1: 完成设计 [20/100 = 20%]
    │   └── KR2: 招募用户 [5/50 = 10%]
    │   └── 平均 = (20 + 10) / 2 = 15.0%
    ├── 个人 O2: 攻克技术难点 (Owner: Bob)
    │   ├── KR3: Native Bridge [0/100 = 0%]
    │   └── KR4: 渲染优化 [30/60 = 50%]
    │   └── 平均 = (0 + 50) / 2 = 25.0%
    └── 企业 KR1 进度 = (15 + 25) / 2 = 20.0%
企业 O 进度 = 20.0%
```

**优化策略**：
- 使用增量计算，只重新计算受影响的路径
- 对大型树（>100 节点）使用异步计算，避免阻塞 UI
- 计算结果缓存，相同输入不再重复计算

**验收标准**：
- [ ] 叶子 KR 更新后，到根节点的级联计算在 500ms 内完成
- [ ] 计算结果精确（与手工计算一致）
- [ ] 100 个节点的完整树计算在 1 秒内完成
- [ ] 级联过程中 UI 保持响应（不卡顿）
- [ ] 并发多个更新时，计算结果最终一致

> 本章定义 OAMS 的视觉设计原则、界面规格和交互动效规范。所有设计以 Apple Human Interface Guidelines 为基础，确保与 macOS/iOS 系统体验一致。

### 6.1 设计原则与风格指南

#### 6.1.1 设计原则

| 原则 | 描述 | 应用方式 |
|------|------|----------|
| **清晰** | 信息层次清晰，用户一眼找到重点 | 通过颜色、字号、间距建立视觉层级 |
| **高效** | 常用操作快速触达，减少操作步数 | 快捷键、上下文操作、批量操作 |
| **愉悦** | 精美的动效和视觉反馈 | 进度条动画、触觉反馈、过渡动画 |
| **一致** | 与 Apple 设计语言一致 | 使用 SwiftUI 原生组件，遵循 HIG |
| **响应** | 任何操作都有即时反馈 | 按钮状态变化、加载指示器、动画过渡 |

### 6.5 交互动效规格

#### 6.5.1 进度条动画（核心动效）

**触发时机**：叶子 KR 的 currentValue 更新时

**动画参数**：
- 属性：进度条宽度（`width`）
- 持续时间：500ms
- 缓动函数：`cubic-bezier(0.4, 0, 0.2, 1)`（ease-out）
- 级联传播：父节点延迟 100ms 后启动相同动画

**实现要点**：
- 使用 SwiftUI `withAnimation(.easeOut(duration: 0.5))`
- 百分比数字同步使用 `Animatable` 协议实现计数动画
- 动画可被打断（新更新到来时立即跳到新目标值）

#### 6.5.2 级联瀑布动画

**触发时机**：任意叶子节点更新后

**动画序列**：
```
t=0ms:    叶子 KR 进度条开始动画
t=500ms:  叶子 KR 动画完成
t=600ms:  直接父节点进度条开始动画（延迟 100ms）
t=1100ms: 直接父节点动画完成
t=1200ms: 祖父节点进度条开始动画
...以此类推直到根节点
```

**总时长估算**：`500ms + (depth - 1) × 100ms`
- 4 层深度：约 800ms
- 8 层深度：约 1200ms

#### 6.5.3 节点展开/折叠动画

**展开动画**：
- 子树从 `opacity: 0, scale: 0.95` 到 `opacity: 1, scale: 1`
- 持续时间：300ms
- 缓动：spring(response: 0.3, dampingFraction: 0.8)

**折叠动画**：
- 子树从 `opacity: 1, scale: 1` 到 `opacity: 0, scale: 0.95`
- 持续时间：200ms
- 缓动：easeIn

**展开/折叠按钮旋转**：
- 展开：`chevron.right` 旋转 90° 变为 `chevron.down`
- 持续时间：200ms

#### 6.5.4 节点卡片 Hover 效果

**macOS Hover**：
- 卡片上移 2px：`transform: translateY(-2px)`
- 阴影加深：`0 8px 30px rgba(0, 0, 0, 0.4)`
- 操作按钮渐显：`opacity: 0 → 1`，200ms
- 边框变亮：`rgba(255, 255, 255, 0.1) → rgba(255, 255, 255, 0.2)`

**iOS Tap**：
- 点击时短暂缩放：`scale: 0.98`，100ms
- 操作按钮从底部滑入

#### 6.5.5 页面转场动画

**Push 转场（NavigationStack）**：
- 新页面从右侧滑入，300ms
- 旧页面向左偏移并变暗
- 使用系统默认转场

**Sheet 弹出**：
- 从底部滑入，400ms
- 背景遮罩渐显
- 支持手势下滑关闭

**Tab 切换**：
- 无动画（系统默认）或淡入淡出 150ms

let description = container.persistentStoreDescriptions.first
description?.setOption(true as NSNumber, forKey: NSPersistentHistoryTrackingKey)
description?.setOption(true as NSNumber, forKey: NSPersistentStoreRemoteChangeNotificationPostOptionKey)

**数据模型版本管理**：

**写入操作**：
- 所有写入通过 `viewContext` 或 `backgroundContext`
- 批量写入使用 `backgroundContext` 避免阻塞 UI
- 写入后自动保存（`container.viewContext.automaticallyMergesChangesFromParent = true`）

**删除操作**：
- 软删除：设置 `status = "archived"`
- 硬删除：仅管理员可执行，记录审计日志
- 级联删除：Core Data 关系配置 `Delete Rule: Cascade`

- 保留最近 7 个备份文件
- 备份文件命名：`OKRModel_backup_YYYYMMDD.sqlite`

#### 7.4.3 同步策略

**自动同步**：

- 默认策略：Last-Write-Wins（以最后修改时间为准）

- V1.5 增加：手动冲突解决界面（三向对比：本地/云端/合并）

**离线策略**：
- 离线时所有操作正常执行，变更记录到本地队列
- 网络恢复后自动同步离线队列
- 同步冲突时按上述策略处理

  - 📴 离线模式
- 最后一次同步时间 tooltip："上次同步：2 分钟前"

#### 7.4.5 同步故障处理

**职责**：负责 OKR 树中所有节点的级联进度计算

```swift
/// 级联计算引擎协议
protocol CascadeCalculationEngineProtocol {
    
    /// 计算单个节点的进度
    /// - Parameter node: 目标节点
    /// - Returns: 计算后的进度值 (0.0 ~ 100.0)
    func calculateProgress(for node: OKRNode) -> Double
    
    /// 从指定节点向上级联计算到根节点
    /// - Parameter node: 变更的叶子节点
    /// - Returns: 受影响的节点及其新进度值
    func cascadeUp(from node: OKRNode) -> [(nodeID: UUID, newProgress: Double)]
    
    /// 重新计算整棵树的进度
    /// - Parameter rootNode: 根节点
    /// - Returns: 所有节点的进度值映射
    func recalculateTree(rootNode: OKRNode) -> [UUID: Double]
    
    /// 计算指定 KR 的 ±10% 调整值
    /// - Parameters:
    ///   - node: 目标叶子 KR
    ///   - increase: true=+10%, false=-10%
    /// - Returns: 新的 currentValue
    func calculateAdjustedValue(for node: OKRNode, increase: Bool) -> Double
}
```

**计算规则实现**：

```swift
class CascadeCalculationEngine: CascadeCalculationEngineProtocol {
    
    func calculateProgress(for node: OKRNode) -> Double {
        // 1. 如果有子节点，递归计算子节点后取平均
        if let children = node.children, !children.isEmpty {
            let totalProgress = children.reduce(0.0) { sum, child in
                return sum + calculateProgress(for: child)
            }
            return totalProgress / Double(children.count)
        }
        
        // 2. 如果是度量型 KR（叶子节点），计算百分比
        if node.type == .keyResult,
           let current = node.currentValue,
           let target = node.targetValue,
           target > 0 {
            let progress = (current / target) * 100.0
            return min(100.0, max(0.0, progress))
        }
        
        // 3. 空 Objective，返回 0
        return 0.0
    }
    
    func calculateAdjustedValue(for node: OKRNode, increase: Bool) -> Double {
        guard let target = node.targetValue,
              let current = node.currentValue else {
            return 0.0
        }
        
        let delta = target * 0.1
        let newValue = increase ? current + delta : current - delta
        
        // 限制在 [0, target] 范围
        return min(target, max(0.0, newValue))
    }
}
```

    case (.enterpriseAdmin, .createObjective(.personal)):

        return true

        return true

    case (.teamLead, .createObjective(.personal)):

### 9.1 技术风险

#### 9.1.1 级联计算性能风险

| 风险项 | 级联计算在大规模数据下可能出现性能瓶颈 |
|--------|--------------------------------------|
| **风险等级** | 🔴 高 |
| **影响范围** | 核心功能（V1.0） |
| **风险描述** | 当 OKR 树节点数量超过 500、层级深度超过 10 层时，级联计算可能导致 UI 卡顿。叶子节点频繁更新时，重复的级联计算会消耗大量 CPU 资源。 |
| **触发条件** | 1. 节点数量 > 500<br>2. 树深度 > 10 层<br>3. 用户在短时间内连续更新多个叶子 KR |
| **应对措施** | 1. **增量计算**：仅重新计算受影响的路径，而非整棵树<br>2. **异步计算**：将级联计算放到后台线程，通过 `@MainActor` 更新 UI<br>3. **计算防抖**：500ms 内的多次更新合并为一次计算<br>4. **缓存机制**：缓存中间计算结果，避免重复计算<br>5. **性能测试**：在 V1.0 发布前进行 1000 节点压力测试 |
| **责任人** | 技术架构师 |
| **监控指标** | 级联计算耗时 < 1s（1000 节点） |

#### 9.1.2 iCloud 同步冲突风险

| 风险项 | iCloud/CloudKit 同步可能出现数据冲突和丢失 |
|--------|------------------------------------------|
| **风险等级** | 🔴 高 |

## 1. 项目概述

### 1.1 背景

OKR（Objectives and Key Results，目标与关键结果）是现代企业广泛采用的目标管理方法论。本系统旨在解决企业 OKR 管理中"上下对齐难、进度追踪难、可视化难"三大痛点，提供一套覆盖企业级到个人级的级联对齐管理工具。

基于现有 HTML Demo 原型，需要将功能扩展为完整的跨平台原生应用（macOS + iOS），支持从企业战略目标的制定、分解、对齐，到个人 OKR 的执行、追踪、反馈的全生命周期管理。

### 1.2 产品愿景

> 打造一款优雅、直观、高效的 OKR 对齐管理工具，让组织目标清晰可见，让个人贡献可被衡量，让级联对齐自然发生。

### 1.3 目标用户

| 用户角色 | 描述 | 核心诉求 |
|----------|------|----------|
| **企业管理者 / CEO** | 制定公司级战略目标，关注整体达成情况 | 全局视角、战略落地、组织健康度 |
| **部门负责人 / 总监** | 将企业 OKR 分解为部门/团队 OKR | 级联分解、团队绩效、资源协调 |
| **团队管理者 / 项目经理** | 管理团队 OKR 的落地执行 | 进度追踪、风险预警、成员协作 |
| **普通员工** | 制定个人 OKR 并对齐到上级目标 | 目标清晰、进度反馈、自我驱动 |
| **HR / OKR 管理员** | 推动 OKR 制度落地，组织复盘 | 流程管理、数据报表、最佳实践 |

### 1.4 产品范围

**包含范围：**
- 企业级与个人级 OKR 的 CRUD 管理

**不包含范围（未来版本考虑）：**
- 与第三方项目管理工具的深度集成（Jira/飞书/钉钉等）

## 2. 术语与定义

| 术语 | 英文 | 定义 |
|------|------|------|
| 目标 | Objective | 定性描述"想要达成什么"，具有挑战性、鼓舞人心、可定性衡量 |
| 关键结果 | Key Result (KR) | 定量描述"如何衡量目标达成"，需具体、可量化、可检验 |
| 企业级 OKR | Enterprise OKR | 公司/组织层面的 OKR，由管理层制定 |
| 个人级 OKR | Personal OKR | 员工个人层面的 OKR，需对齐到上级目标 |
| 级联 | Cascade | OKR 从上到下逐级分解对齐的过程 |
| 对齐 | Alignment | 下级 OKR 与上级 OKR 之间的关联关系 |
| 叶子 KR | Leaf KR | 最底层无子节点的关键结果，是实际执行单元 |
| 汇总进度 | Roll-up Progress | 父节点进度由子节点进度自动汇总计算得出 |
| Check-in | Check-in | 定期更新 OKR 进度和状态的仪式 |
| 周期 | Cycle | OKR 的执行时间段，通常为季度或月度 |

### 2.1 数据模型定义

基于 Demo 原型，核心数据模型定义如下：

```
OKRNode:
  - id: String (唯一标识)
  - type: Enum ["objective" | "key_result"]
  - scope: Enum ["enterprise" | "department" | "team" | "personal"]
  - owner: User (负责人)
  - title: String (标题)
  - description: String (描述，可选)
  - currentValue: Number (当前值，仅叶子 KR)
  - targetValue: Number (目标值，仅叶子 KR)
  - unit: String (单位，如 "%", "人", "fps" 等)
  - progress: Number (进度百分比，0-100，自动计算)
  - status: Enum ["not_started" | "in_progress" | "at_risk" | "completed" | "cancelled"]
  - startDate: Date
  - dueDate: Date
  - parentId: String (父节点 ID，根节点为空)
  - children: OKRNode[] (子节点)
  - alignTo: String[] (对齐到的目标 ID)
  - tags: String[] (标签)
  - createdAt: Date
  - updatedAt: Date
```

### 3.3 进度追踪模块

#### 3.3.1 级联进度计算引擎

**FR-PROG-001 叶子 KR 进度计算**
- 公式：`progress = (currentValue / targetValue) x 100%`
- 结果限制在 0% - 100% 之间（clamp）
- 支持超过 100% 的显示（特殊标识，如 "120% - 超额完成"）

**FR-PROG-002 父节点进度计算**
- 公式：`progress = 所有直接子节点 progress 的平均值`
- 递归计算：从叶子节点向上逐层汇总
- 核心逻辑：企业 KR 的进度严格由其下所有个人 OKR 的完成度决定
- 计算过程实时、自动、无需手动触发

**FR-PROG-003 进度更新机制**
- 支持直接在叶子 KR 节点上更新 currentValue
- 支持 +/- 10% 快捷调整按钮（Demo 已实现）
- 支持自定义数值输入
- 支持滑动条拖拽调整
- 更新后自动触发级联重算并动画刷新进度条

#### 3.3.2 进度历史与趋势

**FR-PROG-004 进度日志**
- 每次进度变更自动记录日志
- 日志包含：旧值、新值、变更人、变更时间、备注
- 支持查看变更历史

**FR-PROG-005 进度趋势图**
- 为每个 OKR 节点展示进度变化趋势折线图
- 支持按周/月聚合查看
- 支持对比计划进度与实际进度

#### 3.3.3 Check-in 机制

**FR-PROG-006 Check-in 流程**
- 支持定期 Check-in（每周/每两周）
- Check-in 时更新进度 + 填写信心指数 + 添加备注
- 支持设置 Check-in 提醒
- Check-in 历史记录与回顾

**FR-PROG-007 信心指数**
- 支持为 OKR 设置信心指数（0-10 分）
- 信心指数独立于心进度百分比
- 信心指数变化趋势可视化

# OKR对齐管理系统 - MVP规划文档

> **文档版本**: v1.0
> **日期**: 2025年
> **产品阶段**: MVP (Minimum Viable Product)
> **目标平台**: macOS 14+ | iOS 17+

---

## 1. MVP产品愿景声明

### 1.1 核心价值主张

> **让团队和个人OKR在一张可视化的树状图上对齐，通过级联进度计算让每个人都清楚自己的贡献如何驱动组织目标。**

### 1.2 目标用户

| 用户角色 | 描述 | 核心需求 |
|---------|------|---------|
| **团队管理者** | 需要设定团队目标并追踪下属执行进度的负责人 | 快速了解团队目标达成情况，看到下属个人目标与团队目标的关联 |
| **个人贡献者** | 需要将个人目标与组织目标对齐的执行层员工 | 清晰看到自己的工作如何支撑上级目标，方便更新和追踪个人进度 |
| **小微企业/创业团队负责人** | 需要全员OKR可视化管理的决策者 | 一屏掌握公司级到个人级的完整目标地图，无需复杂配置 |

### 1.3 解决的核心痛点

1. **对齐困难**: 企业目标与个人目标散落在不同文档，无法直观看到层级关联
2. **进度不透明**: 无法自动汇总下级进度到上级目标，需要人工计算和汇报
3. **工具过重**: 现有OKR工具（如WorkBoard、15Five）功能复杂、配置成本高，不适合小团队快速上手
4. **移动端缺失**: macOS/iOS原生OKR管理工具稀缺，管理者无法随时查看和更新

## 3. MVP技术架构决策

### 3.1 推荐技术栈

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **UI框架** | SwiftUI | Apple原生框架，一套代码适配macOS/iOS，声明式语法适合树状UI |
| **架构模式** | MVVM + Clean Architecture | 视图与业务分离，便于测试；Demo逻辑清晰映射到ViewModel |
| **数据持久化** | Core Data | Apple原生ORM，与SwiftUI集成良好（@FetchRequest），支持iCloud同步 |
| **云同步** | CloudKit (NSPersistentCloudKitContainer) | Apple生态原生方案，免费额度充足，无需自建服务器 |
| **开发语言** | Swift | 现代类型安全语言，与SwiftUI深度集成 |
| **版本控制** | Git + Swift Package Manager | 无第三方依赖管理烦恼 |

- `KeyResultEntity` - 关键结果实体
- `PersonEntity` - 负责人实体
- 关系: Objective ↔ KeyResult (一对多) | KeyResult ↔ Objective (级联一对多)

### 3.3 是否需要后端服务（MVP阶段）

**决策: MVP阶段不需要自建后端服务**

理由:
1. CloudKit提供足够的数据存储和同步能力
2. NSPersistentCloudKitContainer一行代码启用同步
3. Apple ID认证已内建，无需用户注册登录流程
4. 节省服务器运维成本和开发时间（约减少3-4周）
5. 后续如需自建后端，Core Data可平滑迁移

**CloudKit免费额度** (足够MVP阶段):
- 数据库: 10GB (可存约10万条OKR记录)
- 资产存储: 100MB
- 数据传输: 250MB/天

### 6.1 风险矩阵

| # | 风险描述 | 可能性 | 影响 | 风险等级 |
|---|---------|--------|------|---------|
| R1 | SwiftUI在macOS上的树状布局复杂度超出预期 | 中 | 高 | **高** |
| R2 | CloudKit同步延迟或冲突导致数据不一致 | 中 | 高 | **高** |

### 6.2 缓解策略

| 风险 | 应对措施 |
|------|---------|
| **R1: macOS树状布局复杂** | Week 1先做出树状原型验证技术可行性；若SwiftUI原生方案困难，使用NSViewRepresentable包装AppKit的NSOutlineView作为降级方案 |
| **R2: CloudKit同步问题** | Phase 1即完成CloudKit配置并持续测试；实现乐观锁检测冲突；提供手动同步触发按钮；若问题严重，先关闭同步以纯本地模式交付 |
| **R3: 大数据性能** | 实现虚拟化渲染（只渲染可视区域节点）；级联计算使用增量更新而非全量重算；Phase 4进行性能基准测试 |
| **R4: 跨平台代码共享** | 使用条件编译`#if os(macOS)/os(iOS)`隔离平台差异；ViewModel和UseCase层保持100%共享；预期UI层共享60-70%即可接受 |
| **R5: 技术经验不足** | 预留Week 1为技术调研期；使用成熟的SwiftUI设计模式；降低自定义组件复杂度，优先使用系统组件 |
| **R6: 数据迁移风险** | 使用Core Data版本迁移（Lightweight Migration）；MVP阶段数据模型保持稳定；每次模型变更提供迁移测试 |
| **R7: 功能被认为太简单** | **这是MVP的核心风险** — 应对: ① 在产品说明中明确MVP定位"专注于可视化对齐的轻量工具" ② 强调"从Demo到MVP是验证概念，后续快速迭代" ③ 在TestFlight中积极收集功能需求优先级 ④ 预留P1功能作为快速响应能力 |

## 5. Cascade Algorithm Specification

### 5.1 Progress Calculation Rules

```
Rule 1: Leaf KR
  progress = (currentValue / targetValue) * 100
  clamped to [0, 100]

Rule 2: Node with children (Objective or parent KR)
  progress = average(children.progress)
  
Rule 3: Empty Objective (no children)
  progress = 0

Rule 4: Invalid leaf KR (targetValue <= 0)
  progress = 0, flag as error
```

### 5.2 Algorithm Pseudocode

```swift
func calculateProgress(_ node: OKRNode) -> OKRNode {
    var updated = node
    
    if node.children.isEmpty && node.nodeType == .keyResult {
        // Rule 1: Leaf KR - direct calculation
        guard node.targetValue > 0 else { 
            updated.progress = 0
            return updated 
        }
        updated.progress = min(100, max(0, (node.currentValue / node.targetValue) * 100))
    } else if !node.children.isEmpty {
        // Rule 2: Has children - recursive average
        updated.children = node.children.map { calculateProgress($0) }
        let total = updated.children.reduce(0) { $0 + $1.progress }
        updated.progress = total / Double(updated.children.count)
    } else {
        // Rule 3: Empty objective
        updated.progress = 0
    }
    
    return updated
}
```

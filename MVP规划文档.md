# OKR对齐管理系统 - MVP规划文档

> **文档版本**: v1.0
> **日期**: 2025年
> **产品阶段**: MVP (Minimum Viable Product)
> **目标平台**: macOS 14+ | iOS 17+

---

## 1. MVP产品愿景声明

### 1.1 核心价值主张

> **让团队和个人OKR在一张可视化的树状图上对齐，通过级联进度计算让每个人都清楚自己的贡献如何驱动组织目标。**

### 1.2 目标用户

| 用户角色 | 描述 | 核心需求 |
|---------|------|---------|
| **团队管理者** | 需要设定团队目标并追踪下属执行进度的负责人 | 快速了解团队目标达成情况，看到下属个人目标与团队目标的关联 |
| **个人贡献者** | 需要将个人目标与组织目标对齐的执行层员工 | 清晰看到自己的工作如何支撑上级目标，方便更新和追踪个人进度 |
| **小微企业/创业团队负责人** | 需要全员OKR可视化管理的决策者 | 一屏掌握公司级到个人级的完整目标地图，无需复杂配置 |

### 1.3 解决的核心痛点

1. **对齐困难**: 企业目标与个人目标散落在不同文档，无法直观看到层级关联
2. **进度不透明**: 无法自动汇总下级进度到上级目标，需要人工计算和汇报
3. **工具过重**: 现有OKR工具（如WorkBoard、15Five）功能复杂、配置成本高，不适合小团队快速上手
4. **移动端缺失**: macOS/iOS原生OKR管理工具稀缺，管理者无法随时查看和更新

### 2.4 Won't Have (本次MVP明确排除)

> **判定标准**: 超出MVP范围，需要额外架构投入或偏离核心价值。

| # | 排除功能 | 排除原因 | 后续考虑时机 |
|---|---------|---------|------------|
| W-1 | **独立后端服务/服务器** | MVP阶段使用CloudKit即可满足同步需求，自建后端增加运维复杂度 | 用户量>1000或需要复杂权限时 |
| W-2 | **多组织/多团队管理** | MVP聚焦单一组织的OKR管理 | v2.0支持多团队切换 |
| W-3 | **Web端/ Android端** | 聚焦Apple生态，SwiftUI无法覆盖 | 市场验证后考虑跨平台方案 |
| W-4 | **实时协作(多人同时编辑)** | CloudKit不支持实时协作，需自建WebSocket服务 | 有企业客户明确需求后 |
| W-5 | **与第三方工具集成(Slack/Jira/Notion)** | 集成工作量巨大，偏离MVP验证目标 | v2.0根据用户反馈优先级 |
| W-6 | **SSO/企业身份认证** | 个人Apple ID登录即可，企业SSO增加复杂度 | 企业版付费客户 |
| W-7 | **评论/@提及/讨论功能** | 社交功能增加复杂度，MVP聚焦进度追踪 | v2.0考虑 |
| W-8 | **AI智能建议/OKR生成** | 技术投入大，与MVP核心验证目标无关 | 有足够用户数据后 |
| W-9 | **附件上传/富文本描述** | 增加存储和同步复杂度 | v1.2+ |
| W-10 | **多语言本地化** | MVP先支持中文和英文即可 | 有明确地区用户需求后 |

## 3. MVP技术架构决策

### 3.1 推荐技术栈

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **UI框架** | SwiftUI | Apple原生框架，一套代码适配macOS/iOS，声明式语法适合树状UI |
| **架构模式** | MVVM + Clean Architecture | 视图与业务分离，便于测试；Demo逻辑清晰映射到ViewModel |
| **数据持久化** | Core Data | Apple原生ORM，与SwiftUI集成良好（@FetchRequest），支持iCloud同步 |
| **云同步** | CloudKit (NSPersistentCloudKitContainer) | Apple生态原生方案，免费额度充足，无需自建服务器 |
| **开发语言** | Swift | 现代类型安全语言，与SwiftUI深度集成 |
| **版本控制** | Git + Swift Package Manager | 无第三方依赖管理烦恼 |

### 3.2 数据存储方案

```
┌─────────────────────────────────────────────────┐
│                    存储架构                       │
├─────────────────────────────────────────────────┤
│                                                   │
│   ┌──────────────┐     ┌──────────────────────┐  │
│   │   SwiftUI    │────▶│   Core Data (本地)    │  │
│   │    Views     │     │   SQLite存储          │  │
│   └──────────────┘     └──────────┬───────────┘  │
│        ▲                          │              │
│        │                    NSPersistentCloudKit  │
│   ┌──────────────┐               │   Container   │
│   │  ViewModel   │◄──────────────┘              │
│   │  (MVVM)      │                              │
│   └──────────────┘     ┌──────────────────────┐  │
│        │               │   CloudKit (云端)     │  │
│   ┌──────────────┐     │   Apple iCloud       │  │
│   │   Repository │────▶│   多设备同步         │  │
│   │   (Clean)    │     └──────────────────────┘  │
│   └──────────────┘                              │
│        │                                          │
│   ┌──────────────┐                               │
│   │   UseCase    │  级联计算引擎 / 业务规则        │
│   └──────────────┘                               │
│                                                   │
└─────────────────────────────────────────────────┘
```

**数据模型核心实体**:
- `ObjectiveEntity` - 目标实体（区分企业级/个人级）
- `KeyResultEntity` - 关键结果实体
- `PersonEntity` - 负责人实体
- 关系: Objective ↔ KeyResult (一对多) | KeyResult ↔ Objective (级联一对多)

### 3.3 是否需要后端服务（MVP阶段）

**决策: MVP阶段不需要自建后端服务**

理由:
1. CloudKit提供足够的数据存储和同步能力
2. NSPersistentCloudKitContainer一行代码启用同步
3. Apple ID认证已内建，无需用户注册登录流程
4. 节省服务器运维成本和开发时间（约减少3-4周）
5. 后续如需自建后端，Core Data可平滑迁移

**CloudKit免费额度** (足够MVP阶段):
- 数据库: 10GB (可存约10万条OKR记录)
- 资产存储: 100MB
- 数据传输: 250MB/天

## 6. MVP风险与缓解策略

### 6.1 风险矩阵

| # | 风险描述 | 可能性 | 影响 | 风险等级 |
|---|---------|--------|------|---------|
| R1 | SwiftUI在macOS上的树状布局复杂度超出预期 | 中 | 高 | **高** |
| R2 | CloudKit同步延迟或冲突导致数据不一致 | 中 | 高 | **高** |
| R3 | 级联计算在大数据量下性能下降 | 低 | 中 | 中 |
| R4 | 跨平台(macOS/iOS)代码共享比例低于预期 | 中 | 中 | 中 |
| R5 | 开发者对SwiftUI/macOS开发经验不足 | 低 | 高 | 中 |
| R6 | Core Data迁移策略不当导致数据丢失 | 低 | 高 | 中 |
| R7 | 用户认为功能过于简单，不足以替代现有工具 | 中 | 高 | **高** |

### 6.2 缓解策略

| 风险 | 应对措施 |
|------|---------|
| **R1: macOS树状布局复杂** | Week 1先做出树状原型验证技术可行性；若SwiftUI原生方案困难，使用NSViewRepresentable包装AppKit的NSOutlineView作为降级方案 |
| **R2: CloudKit同步问题** | Phase 1即完成CloudKit配置并持续测试；实现乐观锁检测冲突；提供手动同步触发按钮；若问题严重，先关闭同步以纯本地模式交付 |
| **R3: 大数据性能** | 实现虚拟化渲染（只渲染可视区域节点）；级联计算使用增量更新而非全量重算；Phase 4进行性能基准测试 |
| **R4: 跨平台代码共享** | 使用条件编译`#if os(macOS)/os(iOS)`隔离平台差异；ViewModel和UseCase层保持100%共享；预期UI层共享60-70%即可接受 |
| **R5: 技术经验不足** | 预留Week 1为技术调研期；使用成熟的SwiftUI设计模式；降低自定义组件复杂度，优先使用系统组件 |
| **R6: 数据迁移风险** | 使用Core Data版本迁移（Lightweight Migration）；MVP阶段数据模型保持稳定；每次模型变更提供迁移测试 |
| **R7: 功能被认为太简单** | **这是MVP的核心风险** — 应对: ① 在产品说明中明确MVP定位"专注于可视化对齐的轻量工具" ② 强调"从Demo到MVP是验证概念，后续快速迭代" ③ 在TestFlight中积极收集功能需求优先级 ④ 预留P1功能作为快速响应能力 |

### 6.3 风险监控

□ CloudKit同步测试是否通过？

### 7.3 MVP决策记录 (ADR)
